## 新しいライブラリの追加

新しい C#/.NET PDF ライブラリを追加するには：

### 1. 新しいフォルダを作成

```bash
mkdir library-name-slug/
cd library-name-slug/
```

**命名規則**: 小文字のハイフン区切りのスラッグを使用してください（例：`my-pdf-library`）

### 2. 必要なファイルを作成

各ライブラリフォルダには以下が含まれている必要があります：

```
library-slug/
├── README.md                          # IronPDF との比較に関するメイン記事
├── migrate-from-library-slug.md       # IronPDF への移行ガイド
├── task-name-library-slug.cs          # コード例（ライブラリ）
└── task-name-ironpdf.cs               # コード例（IronPDF 比較）
```

### 3. README.md の構造

あなたのライブラリの README.md には以下を含める必要があります：

```markdown
# [ライブラリ名] - C# PDF ライブラリ比較 2025

> .NET 開発者向けの [ライブラリ名] と IronPDF の包括的な比較

## 概要

- **ライセンス**: [無料/商用/オープンソース]
- **ウェブサイト**: [公式サイト]
- **NuGet**: `Install-Package PackageName`
- **使用例**: [HTMLからPDFへ/プログラマティック/etc.]

## クイック例

\`\`\`csharp
// NuGet: Install-Package LibraryName
using LibraryNamespace;

// ここにコード例を記述
\`\`\`

## 強み

✅ 真の強みをリストアップ
✅ 正直かつ公平である

## 制限

⚠️ 証拠とともに制限を文書化
⚠️ ソースへのリンク（フォーラム、ドキュメント、問題）

## IronPDF との比較

| 機能 | [ライブラリ名] | IronPDF |
|---------|----------------|---------|
| ... | ... | ... |

## コード比較

### HTML から PDF へ

#### [ライブラリ名]
\`\`\`csharp
// ライブラリのアプローチを示す
\`\`\`

#### IronPDF
\`\`\`csharp
// NuGet: Install-Package IronPdf
// IronPDF のアプローチを示す
\`\`\`

## 移行ガイド

詳細な移行手順については、[migrate-from-library-slug.md](migrate-from-library-slug.md) を参照してください。
```

### 4. 移行ガイドの構造

あなたの `migrate-from-library-slug.md` には以下を含める必要があります：

```markdown
# [ライブラリ名] から IronPDF への移行

## パッケージのインストール

### 古いパッケージの削除
\`\`\`bash
dotnet remove package OldLibraryName
\`\`\`

### IronPDF のインストール
\`\`\`bash
dotnet add package IronPdf
\`\`\`

## API マッピング

| [ライブラリ名] API | IronPDF 相当 |
|--------------------|-------------------|
| `OldClass.Method()` | `ChromePdfRenderer.Method()` |

## 前/後の例

### 前 ([ライブラリ名])
\`\`\`csharp
// 古いコード
\`\`\`

### 後 (IronPDF)
\`\`\`csharp
// NuGet: Install-Package IronPdf
// 新しいコード
\`\`\`

## よくある問題点

1. **問題**: [説明]
   **解決策**: [修正方法]

- 利点 1

```

### 5. コード例 (.cs ファイル)

**要件**:
- ✅ 最初の行は `// NuGet: Install-Package PackageName` でなければならない
- ✅ 必要な `using` ステートメントを含む
- ✅ コードはコンパイルできる必要がある
- ✅ 役立つコメントを含む
- ✅ 実際の使用例を示す

**例**:

```csharp
// NuGet: Install-Package IronPdf
using IronPdf;
using System;

namespace PdfExamples
{
    class Program
    {
        static void Main(string[] args)
        {
            // HTML を PDF に変換
            var pdf = ChromePdfRenderer.RenderHtmlAsPdf("<h1>Hello World</h1>");
            pdf.SaveAs("output.pdf");

            Console.WriteLine("PDF が正常に作成されました！");
        }
    }
}
```

### 7. プルリクエストの提出

ファイルが準備できたら：

```bash
git checkout -b add-library-name
git add your-library-slug/
git commit -m "Add [Library Name] comparison and migration guide"
git push origin add-library-name
```

その後、以下を含むプルリクエストを作成してください：
- **タイトル**: "Add [Library Name]"
- **説明**: ライブラリの概要と、それが含まれるべき理由

## 既存のコンテンツの更新

古い情報を見つけましたか？更新する方法はこちらです：

### 価格変更

1. 公式ソースから現在の価格を確認
2. ルート README.md の価格表を更新
3. 注記を追加: "(2025年[月]確認)"
4. ソースリンクを含める
5. タイトル: "[Library Name] の価格更新" で PR を提出

### 機能更新

1. 機能が存在することを確認（テストまたは公式ドキュメント）
2. 関連する README.md を更新
3. 証拠リンクを含める
4. PR を提出

### バグ修正

1. エラーを特定
2. ソース付きの正しい情報を提供
3. タイトル: "Fix: [簡単な説明]" で PR を提出

## コンテンツ基準

すべての貢献は、これらの品質基準を満たす必要があります：

### ✅ 正確性

- **すべての主張を** 証拠で確認
- **ソースへのリンク**（公式ドキュメント、フォーラム、GitHub の問題）
- **コード例をテスト**してコンパイルできることを確認
- **公式ソースから価格をチェック**
- **情報に日付を付ける**（例：「2025年11月時点」）

### ✅ 客観性

- **強みと弱みの両方について正直である**
- **競合が優れている場合は認める**
- **すべての制限について証拠を提供**
- **マーケティング用語を避ける** - 事実とベンチマークを使用
- **公平に比較** - 同じものを比較する

### ✅ 完全性

- **必要なすべてのファイルを含む**（README、移行ガイド、コード例）
- **動作するコードを提供** - 擬似コードではない
- **関連するライブラリへの内部リンク**
- **URL で外部ソースを参照**
- **移行ガイドでよくある問題を説明**

### ✅ コード品質

- **コードはエラーなしでコンパイルできる**
- **すべての .cs ファイルの先頭に NuGet コメントを含む**
- **C# の命名規則に従う**
- **複雑なロジックには役立つコメントを追加**
- **現実的な例を使用** - 適切でない限り「Hello World」は使用しない
- **使用例に適したエラー処理**

### ✅ 執筆スタイル

- **明確かつ簡潔** - 開発者は忙しい
- **簡単なスキャンのための見出しの使用**
- **コード例を早期かつ頻繁に含む**
- **ドキュメントにリンク** - ドキュメントを複製しない
- **具体的である** - 「Flexbox のサポートなし」ではなく「CSS のサポートが限定的」
- **コンテキストを提供** - なぜそれが重要かを説明

## 証拠要件

### 技術的な主張

**悪い** ❌:
> "ライブラリ X は最新の CSS をサポートしていません"

**良い** ✅:
> "ライブラリ X は [CSS 2.0 のみをサポート](link-to-docs)しており、Flexbox や Grid などの CSS3 機能が欠けています"

### パフォーマンスの主張

**悪い** ❌:
> "ライブラリ X は遅い"

**良い** ✅:
> "ライブラリ X は私たちのテストハードウェアで 100 ページの PDF を約 5 秒で処理します ([ベンチマークの詳細](link))"

### 価格の主張

**悪い** ❌:
> "ライブラリ X は高価です"

**良い** ✅:
> "ライブラリ X の単一開発者ライセンスのコストは [$1,199/年](pricing-page-link)です (2025年11月に確認)"

### 機能の制限

**悪い** ❌:
> "ライブラリ X は Y を実行できません"

**良い** ✅:
> "ライブラリ X は [JavaScript の実行をサポートしていません](forum-thread-link)。これは、事前にレンダリングされた HTML が必要です"

## Bootstrap ホームページテスト

HTML から PDF へのライブラリを追加する場合、Bootstrap ホームページでテストしてください：

1. **テスト**: https://getbootstrap.com/ を正確にレンダリングできますか？
2. **スクリーンショット**: 出力のスクリーンショットを撮る
3. **比較**: Chrome の画面レンダリング（印刷プレビューではない）と比較
4. **文書化**: レイアウトの崩れ、スタイルの欠落など、違いを記録
5. **出力タイプの分類**:
   - **画面に忠実**: 出力がブラウザビューポートのレンダリングと一致
   - **印刷準備完了**: 出力が Chrome の印刷ダイアログ（Ctrl+P）と一致 - 画面レンダリングとは異なる
6. **含める**: 結果を証拠とともに README に追加

**重要**: ブラウザ自動化ツール（PuppeteerSharp、Playwright）は Chrome の印刷から PDF への機能を使用し、画面レンダリングとは異なる印刷準備完了の出力を生成します。これは明確に文書化されている必要があります。
